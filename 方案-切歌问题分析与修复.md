# 切歌功能问题分析与修复方案

> **状态：✅ 已实施** | 实施日期：2026-06-04

## 1. 问题现象

用户播放本地歌曲后说"切歌"，出现以下矛盾行为：

| 实际发生 | 预期 |
|---------|------|
| TTS 播报"好的，已切换到下一首" | ✅ 正确 |
| 紧接着 TTS 播报"当前没有正在播放的歌曲，你想听什么？" | ❌ 矛盾 |
| 前端实际没有切歌 | ❌ 核心问题 |

日志：
```
路由决策: path=xiaoai, reason=音乐:next
切歌请求但无播放缓存，提示用户     ← 问题在这
Edge TTS 合成成功 (16272 bytes)    ← "没有播放缓存"的TTS
Edge TTS 合成成功 (24480 bytes)    ← "已切换到下一首"的TTS
```

## 2. 根因分析：完整调用链

### 2.1 代码路径追踪

用户说"不要这一首，切歌"时，完整的调用链：

```
用户输入
  → classify("不要这一首，切歌")
    → RouteDecision(path="xiaoai", reason="音乐:next", music_action={"action":"next"})
```

**路径 A：xiaoai_service 生成回复（第1002-1019行）**
```python
# 第1002-1014行
if decision.path == "xiaoai":
    result = xiaoai_execute(..., music_action=decision.music_action)
    # xiaoai_service.py 第43-53行:
    #   action = "next" → reply = "好的，已切换到下一首"
    #   handled = True

    # 第1012-1014行：立即 TTS 播报"好的，已切换到下一首"
    await ws.send_json({"type": "done", ... "reply": result["reply"]})
    asyncio.create_task(_maybe_tts(ws, result["reply"], "xiaoai"))  # ← TTS 1
```

**路径 B：_send_music_control 执行音乐动作（第1015-1016行）**
```python
    # 第1016行
    if result.get("music_action"):
        await _send_music_control(ws, result["music_action"])
```

旧代码在 `_send_music_control` 中（已被修改前的版本）：
```python
# 旧代码：第219-229行
if action in ("next", "prev"):
    if not _search_cache:              # ← 问题根源
        msg = "当前没有正在播放的歌曲，你想听什么？"
        await ws.send_json({"type": "chat_error", "message": msg})
        asyncio.create_task(_maybe_tts(ws, msg, "music"))  # ← TTS 2（矛盾！）
        return
    await ws.send_json({"type": "music_control", "action": action})
```

### 2.2 根因总结

**根因 1：`_search_cache` 未被本地播放填充**

`_search_cache` 只在外部搜索（Jamendo/Pixabay/网易云）命中时填充，本地歌曲播放和歌单播放都不会填充它。因此只要播放的是本地歌曲，"切歌"就会被阻断。

```python
# _search_cache 被填充的位置（仅限外部搜索）：
# 第288行: _search_cache[clean_query] = jamendo_songs[:3]   ← Jamendo
# 第302行: _search_cache[...] = pixabay_songs[:3]           ← Pixabay
# _announce_and_play() 和 _build_local_song() 永远不会填充 _search_cache
```

**根因 2：时序矛盾——TTS 在动作验证之前就已播报**

`xiaoai_execute()` 在**不知道**音乐动作能否成功的情况下就返回了成功回复。`_send_music_control()` 的失败发生在 TTS 播报之后，用户听到两个矛盾的消息。

**根因 3：`_search_cache` 的设计已过时**

`_search_cache` 最初设计目的是让后端在用户"切歌"时提供新的搜索结果。但在当前架构中：
- 前端有自己的队列管理（`queueRef`、`builtinQueueRef`）
- 前端的 `next()` 直接在自己队列内推进
- 后端发送 `{"type":"music_control","action":"next"}` 时从不附带歌曲数据
- 所以 `_search_cache` 存在与否对实际切歌行为**毫无影响**

### 2.3 `_search_cache` 的其他使用者

全局搜索 `_search_cache` 的所有引用：

| 位置 | 用途 |
|------|------|
| 第114行 | 声明：`_search_cache: dict[str, list[dict]] = {}` |
| 第288行 | 写入：Jamendo 搜索结果缓存 |
| 旧第220行 | 读取：next/prev 时检查是否存在（**已删除**） |
| ? | `cache_engine.py` 中可能有引用 |

`_search_cache` 的唯一消费者就是已被修复的 next/prev 检查。它是一个**已死代码**（在修复后完全没有任何代码读取它）。

## 3. 当前修复状态

当前磁盘上的 `main.py`（第218-223行）已经包含了一个修复：

```python
# ── 切歌/上一首：直接由前端队列处理 ──
if action in ("next", "prev"):
    # 前端已有歌曲队列（播放器/歌单），直接发指令让前端推进。
    # 如果前端没歌曲，next() 无害（队列空则不操作/播默认曲目）
    await ws.send_json({"type": "music_control", "action": action})
    return
```

**这个修复移除了 `_search_cache` 检查，直接转发所有 next/prev 到前端。**

但用户反馈问题"依然存在"，可能原因：
- 服务器进程未重启，运行的是旧代码
- 或者只是把旧日志发出来希望根本性解决问题

## 4. 残留问题（仅靠上述修复不足以解决）

即使后端不再阻断 next/prev，仍有以下问题：

### 4.1 TTS 承诺与实际行为脱节

`xiaoai_execute()` 无条件返回"好的，已切换到下一首"，但前端可能：
- 队列为空 → `next()` 实际什么都没做
- 单曲队列 → `next()` 循环到同一首歌（`(0+1)%1 = 0`）

**用户听到"切了"，实际没变化。**

### 4.2 前端 `next()` 的单曲队列行为

```typescript
// useMusicPlayer.ts
const next = useCallback(() => {
    const q = queueRef.current;
    const idx = currentIndexRef.current;
    if (q.length > 0) {
        const nextIdx = (idx + 1) % q.length;  // 单曲队列：(0+1)%1 = 0，原地踏步
        playIndex(nextIdx);
    }
}, [playIndex]);
```

当只有 1 首歌时，切歌等于重播同一首。

### 4.3 本地单曲播放时队列只有 1 首

当前 `_announce_and_play()` 发送的歌曲数据：
```python
"songs": [song],  # 始终只发 1 首
```

当用户说"播放夜空中最亮的星"且本地命中时，前端收到的 `songs` 只有 1 首。切歌只能在同一首上循环。

## 5. 完整修复方案

### 方案概述

分三个层次修复，从紧急到完善：

### 5.1 第一层：后端阻断修复（已有，需确认生效）

**文件**：`backend/main.py` 第218-223行

```python
if action in ("next", "prev"):
    await ws.send_json({"type": "music_control", "action": action})
    return
```

- 移除 `_search_cache` 检查
- 始终转发 next/prev 到前端
- **状态**：已修改在磁盘，需重启服务生效

### 5.2 第二层：前台播放状态同步（新增）

**问题**：后端不知道前端是否真的有歌曲在播放，TTS 可能说谎。

**方案**：前端在收到 `next`/`prev` 后，若无法执行（队列为空/无歌曲），发送反馈给后端：

**文件**：`frontend/src/hooks/useMusicPlayer.ts`

```typescript
// 修改 handleMusicControl 中的 next/prev case
case 'next': {
    const q = queueRef.current;
    if (q.length > 0) {
        next();
    }
    // 无论队列是否为空都执行 next()（它内部有空队列保护）
    // 但额外：如果是歌单模式且只有1首，切歌=重播同一首
    // 这本身不是问题，用户在歌单里切歌会自然轮转
    next();
    break;
}
case 'prev': {
    prev();
    break;
}
```

> 当前前端行为已经是安全的（空队列时 next/prev 不报错），不需大改。

### 5.3 第三层：本地播放时发送完整本地歌单（核心优化）

**问题**：`_announce_and_play()` 每次只发 1 首歌，切歌无意义。

**方案**：当本地歌单命中时，将整个本地歌单作为队列发给前端。

**文件**：`backend/main.py`

修改 `_announce_and_play()` 或新增一个"播放本地歌曲+队列"的函数：

```python
async def _announce_and_play_with_context(ws, song: dict, all_songs: list[dict] = None):
    """播放歌曲，同时发送完整可切歌队列"""
    songs_payload = []
    if all_songs:
        # 将当前命中的歌曲放在第一位，其余随机排列
        others = [s for s in all_songs if s["filename"] != song.get("filename", "")]
        import random
        random.shuffle(others)
        ordered = [song] + [_build_local_song(s) for s in others]
        songs_payload = ordered
    else:
        songs_payload = [song]
    
    await ws.send_json({
        "type": "music_control", "action": "play",
        "song_id": song.get("song_id", ""),
        "song_name": song.get("song_name", ""),
        "singers": song.get("singers", ""),
        ...
        "songs": songs_payload,  # ← 完整队列，不只 1 首
    })
```

当用户播放"夜空中最亮的星"时，前端会收到完整本地歌单（7 首），切歌时可以在所有本地歌曲间轮转。

**但需要权衡**：
- 优点：切歌体验好，可以在本地歌曲间自由切换
- 缺点：用户可能只想听指定歌曲，不希望自动播放其他歌曲
- 折中：歌单模式（playlist）发送完整歌曲列表，单曲搜索保留单曲行为，但单曲时"切歌"=随机播放另一首本地歌曲

### 5.4 TTS 时序修复（低优先级）

`xiaoai_execute()` 的 `reply` 在知道音乐动作结果前就生成了。但这个问题影响不大，因为：
- 修复后 next/prev 不会被阻断，TTS 说的"切歌"确实被执行了
- 如果队列为空，前端静默忽略，用户听到"切歌"但没变化——这种边缘情况很少见

**如要彻底修复**：
- 在 `_send_music_control` 成功后发回确认，让前端触发新的 TTS
- 或让 `xiaoai_execute` 不生成 music 相关的 reply，改为由 `_send_music_control` 生成

## 6. 推荐实施顺序

| 优先级 | 步骤 | 说明 |
|--------|------|------|
| P0 | 确认第5.1节修复已生效 | 重启后端服务，验证 next/prev 不再被阻断 |
| P1 | 清理 `_search_cache` 死代码 | 删除声明和所有填充代码（约4处） |
| P1 | 本地播放时发送完整歌单 | 修改 `_announce_and_play`，播放本地歌曲时附带全部本地歌曲作为队列 |
| P2 | 前端切歌增强 | 当切歌但队列只有1首时，可随机选取本地歌曲继续播放 |

## 7. 待讨论问题

1. **单曲搜索 vs 全歌单队列**：用户说"播放夜空中最亮的星"时，是否应该同时把其他本地歌曲也放入队列？还是保持只播放这一首，切歌=随机换一首？

2. **`_search_cache` 是否彻底删除**：该缓存当前已无任何消费者，是否完全移除相关代码？

3. **TTS 承诺问题**：是否需要让后端感知前端播放状态（如前端发送"我在播放"的心跳）？还是接受当前"乐观播报"的设计（先说有切歌，实际由前端尽力而为）？
