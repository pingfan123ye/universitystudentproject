网页端 AI 语音交互窗口 —— 技术设计与实施方案
1. 项目概述
构建一个纯网页端的 AI 语音交互中枢，用户通过麦克风说话，系统自动：

将语音实时转为文字；

智能决策该请求属于“电脑操作/编程任务”（调用 Claude Code 执行）、“通用对话/思考”（调用大模型生成回复）或“智能家居控制”（直接调用小爱服务执行设备指令）；

对高频非编程指令自动缓存，避免重复调用大模型，实现快速响应。

系统无需任何实物智能音箱或传感器，所有逻辑均在网页前端 + 本地后端（或边缘服务）协同完成，可直接在个人电脑上运行，同时作为智能家居控制入口和编程助手。

2. 总体架构
text
[浏览器网页]
   ↕ 麦克风语音输入 (Web Speech API / 录音上传)
   ↕ 文本展示 & 语音合成输出 (TTS)
[后端服务 (本地 Python/Node.js)]
   ├─ 语音转写服务 (Whisper/ASR)
   ├─ 智能路由分发器
   │    ├─ Claude Code 执行器
   │    ├─ 大模型调用模块 (云端 API 或本地模型)
   │    └─ 小爱服务调用模块 (mi-gpt / 米家 API)
   ├─ 高频指令缓存引擎
   └─ 会话与日志记录
关键原则：

前端只负责交互与展示，所有智能决策在后端完成，保持架构清晰。

路由逻辑在文本转写后立即触发，根据关键词、意图分类和缓存命中情况选择执行路径。

小爱服务调用通过 MiGPT 或直接米家 API 完成，与智能音箱解耦。

层次	组件	推荐技术	说明

4. 核心功能模块设计
4.1 语音输入与转写
前端方案：使用浏览器 webkitSpeechRecognition（持续识别）或 MediaRecorder 录制音频，通过 WebSocket 发送到后端，后端用 Whisper 实时转写。

降噪与断句：前端使用 hark 或音量检测自动截取有效语音片段，减少无效处理。

转写文本展示：实时逐字出现在聊天框，并在识别完成后自动发送给路由模块。

4.2 智能路由分发器
路由流程图：

text
[转写文本]
   │
   ├─ 包含编程前缀 且 含关键词“claude”/“工作助手”？
   │    └─ 是 → 调用 Claude Code 执行器
   │
   ├─ 高频指令缓存命中？
   │    └─ 是 → 直接返回缓存的小爱响应，触发小爱执行
   │
   └─ 否则 → 发送至大模型，由大模型:
        ├─ 生成文本回复
        └─ 若需控制家居，输出结构化设备指令 → 小爱执行
4.2.1 编程任务识别规则
前缀触发词（识别后整句送 Claude Code）：

“帮我写代码”

“运行程序”

“执行任务”

“Claude，… ”

“工作助手，…”

实现：正则匹配或关键词表，优先级最高，一经命中直接绕过其他路径。

4.2.2 高频指令缓存
触发条件：同一非编程类指令（标准化后文本）连续出现 ≥3 次。

缓存内容：指令文本（精确/模糊）、对应的大模型响应文本、小爱设备调用动作序列（JSON）。

匹配方式：对输入指令做标准化（去语气词、同义词替换），计算哈希或编辑距离，若相似度 >95% 且命中缓存，则直接返回。

缓存管理：

持久化到 SQLite/JSON 文件。

提供前端界面可查看、删除缓存。

若长时间未触发（如7天），自动失效。

4.2.3 大模型通用处理
对于非编程、非缓存命中的输入，组装提示词发送给大模型。

提示词中注入当前可用的家庭设备列表、用户记忆（可选），要求大模型以 JSON 格式输出：

json
{
  "reply": "亲切的回复文本",
  "device_actions": [
    {"device": "living_light", "action": "on"},
    ...
  ]
}
若 device_actions 不为空，则调用小爱服务执行；同时将 reply 返回前端并合成语音。

4.3 Claude Code 集成
目的：让网页对话窗口能够直接命令本地 Claude Code 完成编程和文件操作。

工作流：

前端收到编程意图指令（如“帮我写一个批量重命名的 Python 脚本”）。

后端将该指令与当前工作目录上下文打包，发送给大模型作为协调器（可选，也可直接传原始指令给 Claude Code）。

大模型返回结构化操作命令（或直接生成给 Claude Code 的 prompt），后端将其转化为 claude CLI 参数。

后端在安全沙箱（如 Docker 或受限子进程）中执行 claude -p "..." 或通过 claude SDK，实时获取输出流。

输出流通过 WebSocket 推送到前端聊天框，支持 Markdown 代码块展示。

操作结束后，将结果（成功/失败/生成的文件路径）总结成语音和文本反馈给用户。

安全措施：

限制允许操作的目录（如 ~/workspace）。

禁止执行危险命令（可通过 claude 的权限配置或沙箱阻止 rm -rf 等）。

执行前可在前端显示预览并等待用户确认（可配置）。

4.4 小爱服务直接调用
即使没有物理小爱音箱，我们仍可使用 MiGPT 项目 提供的接口模拟小爱服务，或直接调用米家云 API 控制真实设备（需绑定账号）。演示环境下可使用虚拟设备状态。

小爱服务调用模块：封装成统一接口 execute_xiaoai(device, action, params)。

在编程路径中，若大模型或缓存返回设备指令，即调用此接口。

对于无真实设备的演示，可用虚拟设备层（如 Python 字典）代替，展示状态变化。

5. 前端界面设计
提供单一网页应用，功能分区：

顶部状态栏：连接状态、当前路由模式、缓存命中次数等。

对话窗口（主体）：

用户消息气泡（含转写文本）。

AI 回复气泡（支持 Markdown、代码块）。

Claude Code 执行输出折叠面板。

设备控制卡片（如“卧室灯已打开”）。

语音控制条（底部）：

麦克风按钮（按住说话/点击切换连续收音）。

语音波形动画。

手动输入文本框（作为语音备用）。

侧边栏（可隐藏）：

高频指令缓存列表。

设备状态实时监控（虚拟或真实）。

路由决策日志。

技术实现：

使用 React 或 Vue 构建，配合 wavesurfer.js 展示音频。

通过 WebSocket 或 HTTP Streaming 与后端保持长连接，确保 Claude Code 输出实时展示。

6. 缓存优化详细流程
text
输入: "打开客厅灯"
标准化: "open_living_light"
查询缓存: 命中 → 获取 { reply: "已打开客厅灯", devices: [...], timestamp }
执行: 直接调用小爱接口，将 reply 返回前端，设备状态更新
未命中: 发送给大模型 → 获得结果 → 记录计数
如果同一标准化指令计数 == 3:
    创建缓存条目，保存大模型返回的全部内容
    标记来源为“自动学习”
    通知用户（语音或提示）“我记住了这个快捷指令”
标准化：移除语气词、标点，替换同义短语（如“开灯”/“打开灯”归一化）。

缓存索引：使用哈希表，key 为标准文本。

过期与淘汰：每个缓存条目有 TTL，超过 7 天未使用自动清除；也可用户手动清除。

缓存优先于大模型调用，但优先级低于编程意图。

7. 后端 API 设计（精简）
端点	方法	描述
/api/voice/transcribe	POST	上传音频文件，返回转写文本
/api/chat	POST	发送转写文本，返回决策路径、回复、设备指令等（SSE/WebSocket）
/api/cache	GET	获取缓存列表
/api/cache/{id}	DELETE	删除某条缓存
/api/devices	GET	获取当前设备状态
/api/devices/control	POST	手动控制设备（用于测试）
/api/claude/execute	POST	直接触发 Claude Code（内部调用）
推荐使用 Server-Sent Events (SSE) 或 WebSocket 实现流式响应，尤其当 Claude Code 输出较长时。

8. 开发阶段
阶段	内容	
1. 基础链路	前端录音 → 后端 Whisper 转写 → 大模型回复 → 前端 TTS 播放	
2. 智能路由	关键词匹配、编程意图识别、小爱模拟接口调用	
3. Claude Code 集成	安全执行沙箱、输出流推送、前端展示	
4. 高频缓存系统	指令计数、标准化、缓存存取、Web 管理面板	
5. 前端美化与整合	UI 完善、设备状态卡片、日志可视化	
集成用户记忆系统，实现个性化回复（参考原 MiGPT 记忆方案）。
