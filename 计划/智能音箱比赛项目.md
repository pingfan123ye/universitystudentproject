#MiGPT 双引擎智能音箱系统 —— 项目落地实施方案
##1. 项目目标与范围
将小爱智能音箱升级为具备“本地快速响应 + 云端大模型深度理解”双引擎的下一代家庭智能管家，并在此基础上实现：

自学习标准操作智能体（AutoSOP） ：高频复杂语义→设备组合自动固化为本地命令，降低延迟与成本。

渐进式记忆与主动情境推测（Memory & Proactive Care） ：长期记录用户生活参数，结合传感器主动提供关怀提醒。

大模型驱动的工具调用扩展：在理解复杂意图后，可调度 Claude Code/OpenClaw 完成代码/文件操作等数字任务，以及继续调度小爱本地引擎执行设备联动。

最终交付一套可私有化部署、隐私可控、持续演进的智能家居中枢系统。

##2. 总体架构设计
2.1 系统分层
text
[用户语音] → [小爱音箱] → [MiGPT 核心路由]
├── 路径A: 本地 SOP 引擎 (离线优先)
└── 路径B: 云端大模型推理 (复杂语义)
    ├── 对话生成
    ├── 意图理解 & 设备决策
    ├── 记忆读写 (向量库 + 知识图谱)
    └── 工具调度 (Claude Code/OpenClaw)
        ↕ 设备执行层 (米家 API / 本地 Hub)
        ↕ 传感器数据采集 (Zigbee/BLE/门磁/人体传感器等)

2.2 核心组件
组件	职责	部署位置
MiGPT 路由模块	接收小爱语音转写文本，判断走本地 SOP 还是大模型	本地边缘服务器（树莓派/迷你PC/私有云）
本地 SOP 引擎	存储并匹配已固化的快捷意图，直接生成设备指令	同上
云端大模型服务	处理复杂语义、多轮对话、生成回复与决策	云端 GPU 服务器（可私有化或调用 API）
自学习观察器	监听大模型会话，发现高频模式并沉淀为 SOP	边缘服务器
记忆系统	存储用户参数、事件、习惯，支持向量检索与图谱推理	边缘服务器（轻量级）或云端同步
情境感知引擎	聚合传感器数据、时间、记忆，触发主动提醒规则	边缘服务器
工具调度网关	将大模型输出的结构化动作转换为 Claude Code/OpenClaw/米家 API 调用	边缘服务器 + 云端沙箱
隐私与权限管理	开关控制、数据脱敏、本地存储优先	全局贯穿

##3. 技术选型与关键依赖
模块	推荐技术/方案	说明
小爱音箱接入	MiGPT (mi-gpt)	已选型，通过拦截/代理小爱请求接入自定义后端
语音转文本	小爱本地 ASR + 云端备用 (Whisper)	优先使用小爱自带转写，复杂纠错送 Whisper
大语言模型	Qwen2.5-72B 或 Llama-3-70B (私有化部署)	需支持 function calling 以输出设备指令和工具调用
向量数据库	Chroma / Qdrant	存储记忆片段，用于情境相似度检索
知识图谱	Neo4j (轻量) 或 RDFLib	存储用户实体关系（如“我-通勤时间-30分钟”）
规则引擎	Node-RED / Python rule engine	实现主动提醒的触发条件编排
传感器接入	Home Assistant / Zigbee2MQTT	统一接入门磁、人体、温湿度等传感器
工具调用沙箱	Docker 容器	安全执行 Claude Code/OpenClaw 的文件修改任务
本地SOP存储	SQLite / JSON 文件	少量数据，无需复杂数据库
隐私开关	物理按键 / 语音指令 / 管理后台	满足“随时可关”需求
##4. 分阶段实施计划
###阶段一：基础双通路打通（2-3周）
部署 MiGPT 并验证小爱 → MiGPT → 本地回复/云端回复的通路。

实现意图分类器（基于关键词+轻量模型）区分“简单操作”和“复杂语义”。

简单操作：匹配预定义模板，直接调用米家 API。

复杂操作：发送至大模型，接收结构化输出（回复文本 + 设备指令JSON）。

建立第一版大模型 Prompt，包含当前家庭设备列表、输出格式约束。

测试基础场景：“打开卧室灯”“我今天很累”。

交付物：可演示的双路径对话与设备控制。

###阶段二：自学习标准操作智能体（3-4周）
在大模型处理通路中增加观察器模块，记录每次交互的：原始语音、意图类别、最终执行的设备动作列表、用户确认状态。

聚类分析器：定期（如每天凌晨）分析日志，发现同一意图下≥3次完全相同的设备动作组合。

自动生成 SOP 条目：

json
{
  "trigger": "我好累啊",
  "actions": ["curtain_close", "water_heater_on", "play_white_noise"]
}

同步到本地 SOP 引擎，并标记来源（系统自动学习）。

在前端管理界面展示已学习的 SOP，支持用户查看、修改、删除。

增加“确认学习”机制（可选）：学习到新模式后小爱播报“我发现您常说‘好累啊’就让我关窗帘、开热水器、放白噪音，需要我记住这个习惯吗？”用户确认后生效。

交付物：自动 SOP 生成闭环，本地秒级响应已学习习惯。

###阶段三：渐进式记忆与情境推测（4-6周）
记忆采集：大模型对话中加入信息抽取指令，当用户说出可结构化信息时（如“我上班半小时”“明早七点开会”），抽取为三元组存入知识图谱。示例：(User, commute_time, 30min), (Meeting, start_time, 2026-05-17 07:00)

传感器接入：通过 Home Assistant 接入门磁（卧室门）、人体传感器、窗帘电机状态等。

情境规则配置：触发条件：时间到达会议开始时间 - 通勤时间 - 预留缓冲（如10分钟）时，检测到卧室仍有人体且窗帘关闭。动作：生成自然语言提醒 + 主动询问是否开窗帘、开灯。

主动提醒执行：规则引擎触发后，构建 Prompt 要求大模型生成一句包含当前具体信息的提醒语，并通过小爱 TTS 播放。

记忆检索增强：大模型在生成回复前，可检索相关记忆（如通勤时长、偏好）以个性化回答。

隐私开关：语音指令“关闭智能提醒”立即暂停主动规则触发；同时提供 Web 界面彻底开关。

交付物：基于用户记忆和传感器的主动关怀提醒，可随时关闭。

###阶段四：大模型工具调用扩展（2-3周，可与阶段三并行）
在大模型 function calling 定义中加入新工具：

claude_code_execute(project_path, instruction)：调用 Claude Code 进行代码修改。

openclaw_automate(task_description)：调用 OpenClaw 执行自动化脚本。

安全沙箱：所有代码/文件操作在独立 Docker 容器内进行，目录映射只读或限制写入区域。

用户确认机制：涉及文件修改的操作必须生成预览，小爱播报“即将修改xxx文件，内容为…，确认执行吗？”

执行反馈：操作结果（成功/失败/差异）通过小爱语音反馈，或推送到手机通知。

交付物：通过自然语言实现“帮我优化这个脚本”“新建一个笔记记录今天的会议”等数字操作。

##5. 关键问题与解决方案
5.1 意图分类准确性与边界模糊
问题：用户说“我想放松一下”，既可能是简单开灯，也可能是需要大模型安慰的复杂情感。分类错误会导致体验割裂。

解决方案：

使用三层漏斗分类：第一层关键词匹配（如“打开/关闭”直接进入本地路径）；第二层使用轻量级文本分类模型（DistilBERT）判断复杂度；第三层若置信度不高，默认送大模型处理，由大模型内部判断是否生成简单操作。

用户反馈修正：如果大模型输出极简单动作，可提示“以后直接说打开XX就能快速执行哦”，反向优化分类器。

5.2 自学习 SOP 的误学习与冲突
问题：偶然三次相同操作不一定是习惯，可能误学习；或者习惯改变后旧 SOP 不再适用。

解决方案：

加入置信度评分：观察频率、时间跨度、是否跨天等维度，不止简单计数。

支持 SOP 冷却与衰减：长时间未触发自动失效，通知用户。

所有自动生成的 SOP 均可在 App 中查看和手动删除/编辑，提供完全控制权。

5.3 隐私与数据安全
问题：长期记录生活参数、传感器数据、对话内容涉及高度敏感信息。

解决方案：

边缘优先存储：记忆、SOP、传感器历史全部存本地边缘服务器，云端仅做临时转发大模型推理，且推理请求中脱敏人名、地点（替换为标签）。

端到端加密：语音转写与传感器数据传输加密。

透明的数据审计：提供数据看板，用户可查看系统记录了什么，支持一键清除部分或全部记忆。

物理硬件开关支持（如音箱顶部按键），彻底断开主动推测功能。

5.4 传感器数据不可靠导致误提醒
问题：门磁感应到开门不一定是起床，可能是夜起，导致早上6:30误报睡过头。

解决方案：

多维传感器融合判断：结合人体传感器（床上持续存在）、光线传感器、手机充电状态等，建立“深睡/已醒”模型。

触发前询问而非直接执行：提醒中采用询问句式“需要我帮您拉开窗帘吗？”，而非自动执行。

允许用户静音时段设置：如设定22:00-7:00免打扰，避免深夜误报。

5.5 大模型响应延迟影响体验
问题：云端大模型生成回复+决策可能需2-4秒，用户等待焦虑。

解决方案：

流式返回与语音合成并行：大模型首 token 后即开始 TTS 生成，边生成边播放。

本地 SOP 覆盖 80% 以上日常请求，将大模型调用量降至最低。

使用量化后的小模型（如7B参数）部署在边缘，针对简单情感场景快速响应，极端复杂才调云端大模型。

5.6 工具调用的安全性与误操作
问题：Claude Code 等工具具有强大修改能力，理解错误可能导致删文件或代码损坏。

解决方案：

所有高风险操作（rm、git force push 等）默认拦截，需白名单二次确认。

操作前在沙箱内预演，展示 diff，用户语音/App 确认后才实际写入。

限制可操作目录，禁止访问系统敏感路径。

5.7 小爱音箱硬件限制
问题：小爱音箱本身不开放本地扩展，只能通过 MiGPT 拦截云端对话，可能有不稳定或认证过期问题。

解决方案：

保持 MiGPT 版本更新，监控官方接口变化。

自备备用唤醒方案：在极端情况下可通过 Home Assistant 触发 TTS 主动播报，绕过小爱原生通道。

长期考虑使用 ESP32-S3 等开源语音模块自行搭建高自由度入口，与小爱并行使用。

##6. 产品化与持续运营
用户引导：初次使用时提供“习惯养成向导”，主动询问通勤方式、起床时间等，加速记忆积累。

智能度评分：在 App 中展示“今日为你节省了X次等待”“已自动执行Y条习惯”，增强粘性。

反馈循环：每次主动提醒后提供“有用/没用”快捷评价，持续优化推测模型。
