技术栈重选型方案 —— 无实物智能音箱（网页端AI语音交互中枢）
================================================================

文档目标：对项目的 7 个核心维度进行完整技术选型，每项说明优势、项目适配理由、替代方案对比，
         最终给出文字版架构流程图和分阶段实施建议。


一、前端技术选型
────────────────────────────────────────────────────────────────

选型结论：React 18 + TypeScript + Tailwind CSS + Vite

【优势】
- React 生态最丰富，社区维护的语音/音频组件（wavesurfer.js、react-speech-recognition）直接可用
- TypeScript 提供类型安全，对多路径路由状态管理（小爱/大模型/Claude三条路径）极为重要
- Tailwind CSS 原子化样式，快速构建聊天界面+设备面板双栏布局，无需手写大量 CSS
- Vite 开发服务器毫秒级热更新，开发体验极佳

【为什么适合这个项目】
- 前端需要同时管理 4 种实时状态流：语音识别文本流、大模型回复流、Claude Code 执行输出流、设备状态变更事件
  React 的 useReducer + Context 模式天然适合这种多源事件汇入单一状态树的场景
- Web Speech API 已有成熟的 React Hook 封装（react-speech-recognition），直接对接浏览器的 SpeechRecognition 和 speechSynthesis
- 项目需要右侧设备面板展示虚拟家庭状态（灯、窗帘等），React 组件化非常适合这种可复用的设备卡片 UI

二、后端技术选型
────────────────────────────────────────────────────────────────

选型结论：Python 3.11+ + FastAPI + WebSocket + asyncio

【优势】
- FastAPI 原生支持 WebSocket 和 SSE（Server-Sent Events），这对本项目至关重要 ——
  大模型回复是流式生成的，Claude Code 执行输出也是流式的，都需要长连接推送
- Python 是 AI/ML 生态的母语：Whisper、Ollama 客户端、transformers 库全在 Python 上
- asyncio 异步模型天然适配本项目的 IO 密集型特征（等语音识别、等大模型回复、等设备响应）
- FastAPI 自动生成 OpenAPI 文档，调试阶段极大便利

【为什么适合这个项目】
- 后端核心逻辑是"接收文本→路由判断→调用外部服务→流式返回"，每个环节都是 IO 操作。
  FastAPI 的 async/await 可以让一个线程同时处理多个用户的多个等待，无需复杂的线程池
- 需要调用的所有外部服务（小爱/MiGPT API、Ollama、Claude CLI）都有成熟的 Python 客户端
- 虚拟设备层（VirtualHome、VirtualSensorHub）用 Python 字典模拟，与真实米家 API 接口一致，
  后续接入真实设备时切换成本为零

三、语音处理技术选型
────────────────────────────────────────────────────────────────

语音处理分两条链路：ASR（语音→文字）和 TTS（文字→语音）。

3.1 ASR（语音识别）

选型结论：Web Speech API（主力）+ Faster-Whisper（离线备胎）

【Web Speech API —— 主力方案】
优势：
- 浏览器内置，零安装、零延迟、完全免费
- Chrome/Edge 对中文识别准确率已相当高（基于云端服务）
- 持续识别模式（continuous: true）天然适合对话场景

适配理由：
- 本项目本身就是"网页端"语音交互，浏览器是唯一入口，Web Speech API 是零成本首选
- 用户无需配置任何 API key，打开网页就能用
- 识别结果通过回调实时返回，直接驱动前端文本展示

限制：
- 需要网络连接（浏览器把音频发到 Google/微软云端识别）
- 断网不可用

【Faster-Whisper —— 离线备胎方案】
优势：
- 基于 OpenAI Whisper 模型，CTranslate2 加速，CPU 上也能跑
- 中文识别准确率一流，支持中英混合
- 完全本地运行，隐私安全

适配理由：
- 作为 Web Speech API 断网时的 fallback
- 对于需要更高精度的场景（如编程指令含英文术语），可切换至此
- medium 模型在普通笔记本 CPU 上延迟约 1-3 秒，可接受

模型选择：Faster-Whisper medium（~1.5GB，精度/速度最佳平衡点）

【替代方案对比】

│ 方案              │ 问题                      │ 结论                  │
├──────────────────┼──────────────────────────┼──────────────────────┤

│ 纯 Faster-Whisper │ 首次加载慢，吃资源        │ 作为离线备用          │

3.2 TTS（语音合成）

选型结论：Edge TTS（主力）+ Web Speech API speechSynthesis（轻量备用）

【Edge TTS —— 主力方案】
优势：
- 微软 Edge 的免费 TTS 服务，中文语音极其自然（xiaoxiao 发音人）
- 通过 edge-tts Python 库直接调用，返回 mp3 音频流
- 延迟低（~200ms），支持 SSML 标记控制语速、停顿

适配理由：
- 项目要求所有回复"语音播报"，对语音自然度要求高
- 直接在后端生成音频流通过 WebSocket 推送到前端播放，比浏览器自带 TTS 质量高一个档次
- 完全免费，无配额限制

【Web Speech API speechSynthesis —— 轻量备用】
优势：浏览器内置，零延迟，无需后端参与
问题：中文语音较机械，不同浏览器语音质量差异大

使用策略：Edge TTS 超时或不可用时，自动降级为 speechSynthesis

四、意图路由技术选型
────────────────────────────────────────────────────────────────

选型结论：三阶段混合路由（关键词规则 → 轻量分类器 → LLM 兜底判断）

【第一层：关键词规则引擎（处理 80% 流量）】

规则结构：
- 编程前缀正则：/^(claude|工作助手|帮我写代码|运行程序|执行任务)/
- 小爱能力关键词字典（设备控制、查询、工具、场景四类）
- 匹配优先级：编程 > 小爱 > 其余走大模型

优势：零延迟（<1ms），100% 可解释，易于调试和扩展
适配理由：项目中定义的"小爱基础功能清单"天然适合关键词匹配

【第二层：轻量文本分类器（处理歧义）】

选型：sentence-transformers + 预训练中文模型（如 bge-small-zh，~100MB）

使用场景：当规则匹配置信度不高时（如"帮我把房间弄得舒服一点"——
到底是设备控制还是闲聊？），用嵌入向量计算与各意图簇的余弦相似度

优势：本地运行，<10ms 推理，能捕捉语义而非仅关键词
适配理由：填补纯规则的盲区，但又不至于每条指令都调大模型做分类

【第三层：LLM 兜底判断（处理复杂歧义）】

仅当前两层都无法确定意图时才调用大模型做最终判断。
此层调用计入大模型调用次数，但不计入高频缓存计数。

【替代方案对比】
┌──────────────────────┬──────────────────────────┬──────────────────┐
│ 方案                  │ 问题                      │ 结论              │
├──────────────────────┼──────────────────────────┼──────────────────┤
│ 纯关键词规则          │ 歧义句/长句无法处理       │ 作为第一层        │
│ 纯 LLM 判断           │ 每条指令都调大模型，延迟高 │ 仅作第三层兜底    │
│ 训练专用分类模型(BERT) │ 需要标注数据，维护成本高  │ 第二层用轻量方案  │
│ fastText              │ 中文支持不如 sentence-     │ 备选，也很轻量    │
│                       │ transformers              │                  │
└──────────────────────┴──────────────────────────┴──────────────────┘


五、大模型调用选型
────────────────────────────────────────────────────────────────

选型结论：Ollama + Qwen2.5:7B（本地主力）+ 云端 API 多路 fallback

【本地主力：Ollama + Qwen2.5:7B】

优势：
- Ollama 一键安装运行，GPU/CPU 自适应
- Qwen2.5（通义千问）中文理解能力在开源模型中顶尖
- 7B 模型仅需 ~4GB 显存/内存，绝大多数个人电脑能跑
- 完全本地，隐私安全，零网络延迟（首次 token ~0.5s）
- 支持结构化 JSON 输出模式（ollama 的 format: json），这对大模型返回
  {"reply": "...", "device_actions": [...]} 格式至关重要

适配理由：
- 项目中大模型的职责是"复杂语义理解 + 生成回复 + 输出设备指令 JSON"，
  Qwen2.5 的中文理解和指令遵循能力完全胜任
- 本地运行意味着高频使用无成本，适合频繁对话场景
- Ollama 提供的 REST API 与 FastAPI 后端天然对接

【云端 fallback 链（按优先级降级）】

1. 通义千问 API（qwen-turbo）—— 与本地模型同系，迁移成本最低，中文最优，极便宜
2. DeepSeek API —— 性价比极高，中文能力强
3. Claude API —— 复杂推理最强，作为编程路径的增强选项

触发条件：
- 本地 Ollama 不可用（未启动/显存不足）
- 需要更强推理能力时（用户明确要求"用更聪明的模型"）
- 本地模型返回格式解析失败，自动升级到云端

【替代方案对比】

六、缓存方案选型
────────────────────────────────────────────────────────────────

选型结论：SQLite（持久层）+ 内存 LRU 字典（热缓存层）

【双层缓存架构】

热层（内存 LRU）：
- Python lru_cache 或手动 dict + 有序链表
- 容量：最近 200 条
- 命中延迟：<0.01ms
- 存储：标准化指令文本 → 完整响应 JSON

冷层（SQLite）：
- 表结构：id | normalized_text | reply | device_actions_json | count | created_at | last_hit_at
- 持久化，重启不丢
- 自动清理：last_hit_at 超过 7 天 → 删除
- 冷热交换：热层满时，最久未命中的条目写回 SQLite；冷层命中时提升到热层

【为什么选 SQLite 而不是 Redis】

- 本项目是单用户本地系统，不需要 Redis 的多进程共享能力
- SQLite 零配置、零进程，一个文件搞定，随项目目录走
- Python 内置 sqlite3 模块，零额外依赖
- 对于"几百条缓存 + 偶尔查询"的场景，SQLite 性能完全过剩

【缓存触发条件（与需求文档一致）】
- 仅大模型路径的指令进入计数
- 同一标准化指令累计 ≥3 次 → 写入缓存
- 第 4 次起命中缓存，不再调用大模型
- 小爱路径和 Claude Code 路径均不计入缓存

【替代方案对比】
┌──────────────┬──────────────────────────┬──────────────────────────┐
│ 方案          │ 问题                      │ 结论                      │
├──────────────┼──────────────────────────┼──────────────────────────┤
│ Redis         │ 需额外安装维护进程        │ 单用户场景过度设计        │
│ JSON 文件     │ 并发写不安全，查询需全量  │ 仅开发初期可接受          │
│               │ 加载                      │                          │
│ 纯内存 dict   │ 重启丢失，积累的缓存白费  │ 需配合持久层              │
│ MongoDB       │ 太重                      │ 完全不必要                │
└──────────────┴──────────────────────────┴──────────────────────────┘


七、部署方式选型
────────────────────────────────────────────────────────────────

选型结论：开发期直接运行 / 演示期 Docker Compose / 分发期单包脚本

【阶段一：开发期 —— 直接运行】
- 前端：npm run dev（Vite dev server，端口 5173）
- 后端：uvicorn main:app --reload（端口 8000）
- Ollama：本地服务，端口 11434
- SQLite：文件数据库，零配置

优势：热更新，改代码即时生效，调试方便

【阶段二：演示期 —— Docker Compose】
一个 compose.yaml 编排三个服务：

  services:
    frontend:    # Node + Nginx 静态服务
    backend:     # FastAPI + edge-tts
    ollama:      # Ollama + Qwen2.5 模型

优势：一键启动（docker compose up），环境完全一致，适合演示

【阶段三：分发期 —— 单包脚本】
- Windows：用 PyInstaller 打包后端 + 内嵌前端静态文件 → 单个 .exe
- 或者用 start.bat 脚本：自动检查 Ollama 安装 → 拉取模型 → 启动后端 → 打开浏览器

优势：非技术用户也能用

十、分阶段实施建议
────────────────────────────────────────────────────────────────

采用"每阶段产出可运行、可演示的增量"策略，共 6 个阶段。

────────────────────────────────────────
阶段 0：环境搭建（半天）
────────────────────────────────────────
目标：所有依赖就绪，端到端"hello world"跑通

具体任务：
□ 初始化前端项目：npm create vite@latest frontend -- --template react-ts
□ 安装依赖：tailwindcss, react-speech-recognition, wavesurfer.js
□ 初始化后端项目：FastAPI 骨架 + WebSocket 端点
□ 安装 Ollama + 拉取 Qwen2.5:7B 模型
□ 安装 edge-tts：pip install edge-tts
□ 验证：用户在网页输入文字 → 后端收到 → 调用 Ollama → 返回回复 → 前端展示

验收标准：
- 浏览器打开能看到聊天界面
- 输入"你好"，1-2 秒内显示 AI 回复
- WebSocket 连接状态显示为绿色

────────────────────────────────────────
阶段 1：基础语音链路（1 天）
────────────────────────────────────────
目标：用户能"说话"交互，听到语音回复

具体任务：
□ 前端接入 Web Speech API，实现麦克风按钮 → 按住说话 → 实时转写文字显示
□ 转写完成自动发送到后端
□ 后端接收文本 → 调用 Ollama → 生成回复
□ 回复文本通过 edge-tts 转为音频 → WebSocket 推送前端 → 自动播放
□ 降级处理：edge-tts 不可用时自动切 speechSynthesis

验收标准：
- 点击麦克风说话，文字实时出现在聊天框
- AI 回复文字出现后，自动用中文语音朗读
- 延迟可接受（从说完话到听到回复 < 3 秒）

────────────────────────────────────────
阶段 2：智能路由分发（2 天）
────────────────────────────────────────
目标：三条处理路径全部跑通，意图分类正确

具体任务：
□ 实现关键词规则引擎（第一层路由）
□ 定义编程前缀正则 + 小爱能力关键词字典
□ 实现三条路径：
  - 小爱路径：虚拟设备执行 → 返回模拟回复 → 播报
  - 大模型路径：调 Ollama → 返回 JSON {reply, device_actions} → 分别处理
  - Claude Code 路径：subprocess 调用 claude CLI → 实时输出流转发
□ 实现 VirtualHome 虚拟设备状态管理
□ 前端增加设备面板，虚拟设备状态变化实时更新
□ 实现统一语音播报出口（所有路径结果走同一个 TTS 函数）

验收标准：
- 输入"打开卧室灯" → 走小爱路径 → 设备面板灯图标点亮 → 播报"好的，已打开"
- 输入"我今天心情不好" → 走大模型路径 → 生成安慰回复 → 播报
- 输入"Claude，帮我写个 hello.py" → 走 Claude Code 路径 → 代码实时输出 → 播报结果

────────────────────────────────────────
阶段 3：高频缓存系统（1.5 天）
────────────────────────────────────────
目标：大模型路径的重复指令自动缓存，3 次学习 + 第 4 次命中

具体任务：
□ 实现指令标准化函数（去语气词、标点，同义词映射）
□ 实现计数逻辑：同一标准化指令每次走大模型路径 +1
□ 实现缓存写入：计数达到 3 → 写入 SQLite + 内存热层
□ 实现缓存查询：请求进入大模型路径前先查缓存，命中则直接返回
□ 前端增加缓存管理面板：查看缓存列表、手动删除、显示命中次数
□ 实现 TTL 自动过期（7 天未命中自动清除）

验收标准：
- 同一句话说 3 次，每次都调大模型（计数 1→2→3）
- 第 4 次说同样的话，立即返回缓存结果，前端显示"缓存命中"
- 缓存面板能看到已缓存的指令列表
- 删除某条缓存后，再说该指令，重新调大模型

────────────────────────────────────────
阶段 4：前端美化 + 演示场景串联（2 天）
────────────────────────────────────────
目标：可演示的完整产品，UI 达到展示级别

具体任务：
□ UI 全面美化（聊天气泡、语音波形、设备卡片动画）
□ 实现原需求文档中的 5 个演示场景按钮：
  - "模拟起床"：传感器变化 + 主动提醒触发
  - "模拟回家"：设备联动
  - 等
□ 添加时间模拟滑块（可拖动时间触发情境引擎）
□ 侧边栏：路由决策日志实时显示
□ 响应式适配（笔记本屏幕 1366×768 为主）

验收标准：
- 10 分钟完整演示流程可顺利走完
- UI 无肉眼可见 bug
- 所有动画流畅

────────────────────────────────────────
阶段 5：进阶特性 + 分发（1.5 天）
────────────────────────────────────────
目标：记忆系统、情境引擎、隐私控制

具体任务：
□ 渐进式记忆系统：对话中提取用户信息 → SQLite 存储
□ 情境感知引擎：定时检查传感器 + 时间 + 记忆 → 触发主动提醒
□ 隐私面板：查看/删除全部记忆、缓存、传感器历史
□ 一键开关主动提醒
□ Docker Compose 编排，一键启动全栈
□ 编写 README 和演示脚本

验收标准：
- 系统能记住用户说过的话并适时引用
- 情境引擎能根据时间+传感器状态触发提醒
- 隐私面板可一键清除所有数据

十二、与原方案的对比总结
────────────────────────────────────────────────────────────────

┌──────────┬──────────────────────┬──────────────────────┐
│ 维度      │ 原方案                │ 新方案                │
├──────────┼──────────────────────┼──────────────────────┤
│ 前端      │ Vue/React（未定）     │ React+TS+Tailwind    │
│ 后端      │ Python/Node（待定）  │ FastAPI+WebSocket    │
│ ASR       │ 仅 Whisper           │ 浏览器 API 主力+      │
│           │                      │ Whisper 备用          │
│ TTS       │ 仅 speechSynthesis   │ Edge TTS 主力+        │
│           │                      │ speechSynthesis 备用  │
│ LLM       │ 提及多种未定         │ Ollama+Qwen2.5:7B    │
│           │                      │ + 云端三级 fallback   │
│ 缓存      │ JSON/SQLite          │ 内存LRU+SQLite 双层   │
│ 意图路由  │ 关键词规则           │ 三层混合路由           │
│ 部署      │ 未明确               │ 三阶段递进             │
└──────────┴──────────────────────┴──────────────────────┘

新方案的核心改进：
1. 每一项选型都有明确的"主力+备胎"，确保断网/低配/缺失依赖等异常情况不崩溃
2. 意图路由从单层关键词升级为三层混合，兼顾速度、准确率和覆盖范围
3. TTS 从浏览器内置升级为 Edge TTS，中文语音自然度显著提升
4. 缓存从简单 JSON 升级为内存热层+SQLite 冷层，性能和持久性兼顾
5. 部署方案覆盖开发→演示→分发全生命周期
