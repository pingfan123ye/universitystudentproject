# 语音识别问题分析与优化方案

## 1. 问题概述

| 问题 | 描述 |
|------|------|
| **无实时转写可视化** | 录音期间前端看不到正在识别的文字，只有停止录音后结果才出现 |
| **识别准确率低** | Whisper small 模型在 CPU 上识别效果差，常见误识别多 |
| **简体/繁体混排** | Transcripiton 经常输出繁体字，需强制简体 |

---

## 2. 当前架构分析

### 2.1 音频采集 → 后端转写 → 显示 完整链路

```
用户说话
  ↓
MediaRecorder(audio/webm)       ← 前端录制完整音频
  ↓ (录制结束 / 静音超时)
解码 WebM → PCM → base64
  ↓
WebSocket: {"type":"audio_stream","audio":"<base64>"}
  ↓
后端 main.py 收到
  ↓
whisper_stt.transcribe_audio_base64(audio_b64)
  ↓  ~ 1-5 秒（CPU 推理）
faster-whisper (small 模型, device="cpu", compute_type="int8")
  ↓
stt_corrector.correct(text)      ← 后处理纠错（有限表）
  ↓
WebSocket: {"type":"stt_result","text":"..."}
  ↓
前端 useWebSocket 收到 → setTranscriptionText
  ↓
ChatPanel 显示在输入框
```

### 2.2 核心问题定位

| 环节 | 问题 |
|------|------|
| **录制方式** | `MediaRecorder` + 整段录制 + 整段发送。录音期间没有任何中间结果返回，用户看不到转写过程 |
| **Web Speech API 被弃用** | 代码中保留了 `useSpeechRecognition()`（浏览器自带 API），但注释说"统一使用后端 Whisper"。其实两者可以并存：Web Speech 做实时显示，Whisper 做最终高质量结果 |
| **Whisper 模型太小** | `small` (244MB) 在 CPU 推理精度有限，`large-v3` (~3GB) 效果显著更好，准确率最高 |
| **CPU 推理慢** | compute_type="int8" 已是最快，但 small 模型在非 GPU 环境下仍需 1-5 秒/段 |
| **无简繁转换** | Whisper 训练数据包含简繁混合，输出不稳定。没有强制简体步骤 |
| **纠错表太简陋** | `stt_corrector.py` 只有 ~30 条替换，覆盖率低 |

## 3. 优化方案

### 3.1 实时转写可视化（建议方案 B — 最实用）

**方案对比：**

| 方案 | 做法 | 复杂度 | 效果 |
|------|------|--------|------|
| A. Web Speech 实时 + Whisper 最终 | 录音期间用浏览器内置 API 显示实时文字，停止后 Whisper 覆盖最终结果 | 低 | 实时可见，但 Web Speech 在中国可能连接 Google 失败 |
| B. **分段发送 + 增量返回** | `_startRec` 时每 2 秒 flush 一段音频到后端，Whisper 转写后增量返回，覆盖到输入框 | **中** | **纯本地，不依赖 Google** |
| C. Websocket 流式 | 音频 PCM 流通过 WebSocket 边录边发到后端，Whisper 实时返回 | 高 | 延迟最低，但需改造 Whisper 为流式 |

**推荐方案 B**：修改 `_startRec` 的逻辑，录制时每隔 2 秒截取当前缓冲区的音频，编码为 base64 发送到后端。后端返回 `stt_result`（增量），前端追加到输入框。

### 3.2 准确率提升

| 改进项 | 实现方式 | 预期提升 |
|--------|---------|---------|
| **模型升级** | `small` → `large-v3`（~3GB，首次需下载，加载较慢） | 准确率最高，Whisper 全系列最佳 |
| **语言强制简体** | Whisper 参数 `language="zh"` + `task="transcribe"` 减少繁体 | 减少繁体字 80%+ |
| **OpenCC 简繁转换** | 安装 `opencc-python`，Whisper 输出后强制转简体 | 杜绝繁体残留 |
| **纠错表扩展** | 增加更多常见 Whiper 误识别模式 | 减少特定词错误 |
| **音频质量** | 录音参数调优：16kHz 采样率、单声道、降低压缩率 | 识别基础更干净 |

### 3.3 具体实施清单

#### 后端改动

| # | 文件 | 改动 |
|---|------|------|
| 1 | `whisper_stt.py` | 模型升级 `small` → `large-v3`；加 `language="zh"` 参数 |
| 2 | `whisper_stt.py` | 安装 opencc-python，Whisper 输出后强制简繁转换 |
| 3 | `stt_corrector.py` | 扩充纠错表（增加常见歌名、指令误识别） |
| 4 | `main.py` | 支持增量转写：前端分段发音频，后端返回后合并 |

#### 前端改动

| # | 文件 | 改动 |
|---|------|------|
| 5 | `ChatPanel.tsx` | `_startRec` 每 2 秒 flush 音频片段到后端（通过 `sendAudioChunk`） |
| 6 | `useWebSocket.ts` | `stt_result` 到达时追加到 `transcriptionText`（增量而非覆盖） |
| 7 | `ChatPanel.tsx` | 录音时输入框实时显示 `transcriptionText`（闪烁光标指示进行中） |

### 3.4 增量转写流程图（方案 B）

```
录音开始
  ↓
setInterval(2s) {
  截取当前 2 秒 PCM 片段
  base64 编码
  WebSocket: {"type":"audio_stream","audio":"..."}
  ↓
  Whisper 转写片段 → stt_corrector → OpenCC 简体
  ↓
  WebSocket: {"type":"stt_result","text":"已识别的文字","final":false}
  ↓
  前端追加到 transcriptionText（末尾闪烁光标）
}
  ↓
录音结束 → 发送最后一段（final=true）→ Whisper 最终结果 → 发送到对话
```

## 4. 风险与注意事项

| 风险 | 缓解 |
|------|------|
| large-v3 模型推理较慢（CPU 每段约 5-15 秒） | 分段 2 秒小片段，推理时间可接受；用户感知延迟 ≤2 秒 |
| 增量转写片段拼接效果差（断句问题） | 前端只做追加不覆盖，用户可手动编辑 |
| OpenCC 误转（专有名词） | 只对 Whisper 输出做强制简繁，不影响其他文本 |
| Web Speech API 在中国不可用 | 方案 B 不依赖 Web Speech，纯本地 Whisper |
