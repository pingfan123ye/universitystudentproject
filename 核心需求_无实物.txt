网页端 AI 语音交互窗口 —— 修正方案
核心原则明确
 
 
规则
说明
所有回复必须语音播报
无论是小爱执行反馈、大模型生成回复、缓存命中还是 Claude Code 结果，都必须通过浏览器 TTS 朗读
小爱优先原则
凡是小爱能完成的（开灯、关窗帘、查天气、定闹钟等基础功能），直接调用小爱服务，不经过大模型
大模型兜底
只有小爱处理不了的复杂语义、多轮对话、情感陪伴等才调用大模型
高频缓存仅针对大模型回答
同一大模型指令连续出现 ≥3 次才缓存其响应，小爱调用不做缓存
 
1. 修订后的路由分发逻辑
text
 
[用户语音 → 转写文本]
        │
        ▼
[意图分类器]
        │
        ├── 是编程/Claude 任务？
        │   （含“Claude”/“工作助手”/“帮我写代码”/“运行程序”等前缀）
        │       └─ YES → Claude Code 执行器 → 结果语音播报
        │
        ├── 是小爱可执行的基础指令？
        │   （开灯/关灯、窗帘、空调、热水器、天气、闹钟、播放音乐等）
        │       └─ YES → 直接调小爱服务 → 小爱返回结果语音播报
        │                 （此路径不做缓存、不经大模型）
        │
        └── 其他（复杂语义、闲聊、情感陪伴、需要推理的问题）
                │
                ├── 高频大模型缓存命中？
                │       └─ YES → 返回缓存的大模型响应 → 语音播报
                │
                └── NO → 调用大模型 → 生成回复 → 语音播报
                              │
                              └── 同一指令累计 ≥3 次？
                                      └─ YES → 写入高频缓存
 
 
 
2. 小爱基础功能清单（直接调用，不经过大模型）
维护一份明确的小爱能力表，命中关键词直接走小爱路径：
python
  
XIAOAI_CAPABILITIES = {
    "device_control": ["打开", "关闭", "调高", "调低", "设置温度", "模式切换"],
    "query": ["天气", "时间", "日期", "空气质量", "湿度"],
    "utility": ["闹钟", "提醒", "倒计时", "播放音乐", "暂停", "下一首", "音量"],
    "scene": ["离家模式", "回家模式", "晚安模式", "起床模式"],
}
 
 
判断方法：对转写文本做关键词匹配，命中即发往小爱，返回结果播报后结束。
 
3. 高频缓存（仅大模型路径）
text
缓存对象：
    key: 标准化后的大模型指令文本
    value: {
        "reply": "大模型生成的完整回复文本",
        "device_actions": [...],  # 若大模型同时输出设备指令
        "count": 3,
        "learned_at": "2026-05-18 10:30"
    }

触发条件：
    - 仅当指令走大模型路径时才计数
    - 同一标准化指令累计出现 3 次，自动写入缓存
    - 第 4 次起，匹配缓存直接返回，不再调用大模型

注意：
    - 小爱直接执行的指令，不进入此计数和缓存系统
    - 缓存可手动查看/删除
 
 
 
4. 语音播报统一处理
所有路径的最终回复必须经过同一个 TTS 出口：
python
  
def speak(text: str):
    """统一语音播报入口"""
    # 发送到前端，触发浏览器 speechSynthesis 或后端 TTS 音频流
    frontend_emit("tts", {"text": text})

播报内容来源

高频缓存命中
缓存中的 reply 文本

操作结果摘要（如“已生成 README 文件”）

6. 完整请求示例
示例一：小爱直接执行（不走大模型、不计缓存）
text
 
用户: "打开客厅灯"
  → 意图分类: 小爱可执行（设备控制）
  → 调用小爱服务
  → 小爱返回: "好的，已打开客厅灯"
  → 浏览器播报: "好的，已打开客厅灯"
  → 虚拟设备状态更新
  → 不计入缓存计数
 
 
示例二：大模型路径 + 触发缓存学习
text
第1次用户: "我今天心情不太好，陪我聊聊天吧"
  → 意图分类: 非小爱能力范围
  → 缓存未命中
  → 调用大模型
  → 大模型回复: "听到你心情不好...（安抚话语）"
  → 播报回复
  → 计数: "陪聊天" → 1

第2次用户: "我今天心情不太好，陪我聊聊天吧"
  → 同上，计数 → 2

第3次用户: "我今天心情不太好，陪我聊聊天吧"
  → 同上，计数 → 3
  → 达到阈值，自动写入缓存
  → 用户收到提示: "我记住了这个对话习惯"

第4次用户: "我今天心情不太好，陪我聊聊天吧"
  → 缓存命中！直接返回缓存的回复
  → 播报，不调大模型
 
 
示例三：Claude Code 编程任务
text
用户: "Claude，帮我写一个批量重命名文件的脚本"
  → 意图分类: 编程前缀
  → 直接发 Claude Code 执行器
  → 实时输出执行过程到前端
  → 完成后播报: "已完成，脚本保存在 rename.py"
 
 
 
7. 架构图（修订版）
text
 
[浏览器前端]
  ├─ 麦克风 → ASR 转写
  ├─ 聊天展示区
  └─ TTS 语音播报（统一出口）
        ↕ WebSocket
[后端核心]
  ├─ 意图分类器
  │     ├─ 编程前缀？ → Claude Code 执行器
  │     ├─ 小爱能力范围？ → 小爱服务调用
  │     └─ 其他 → 大模型路径
  ├─ 高频缓存引擎（仅大模型路径）
  ├─ 大模型调用模块
  ├─ 小爱服务封装（MiGPT / 米家 API / 虚拟设备）
  └─ Claude Code 沙箱执行器
 
 
 
8. 对之前方案的主要修改点
 
 
缓存针对所有非编程指令
缓存仅针对大模型路径的指令
小爱调用在缓存/大模型下游
小爱调用独立优先路径，命中即执行，不计入缓存
未强调统一播报
所有路径结果强制走统一 TTS 出口
小爱作为备选
小爱优先，能做的直接做
 
这样整个系统的边界就非常清晰：小爱干粗活（直接执行）→ 不缓存；大模型干细活（复杂理解）→ 重复才缓存；Claude 干代码活（编程任务）→ 实时执行并播报结果。所有输出都要说话。
